Custom submitter factories were threaded through nested submitter resolution, and the CLI strategy schema was widened to accept a custom nested submitter inside both composite submitters (interchain-account and timelock-controller). Previously `getSubmitter` passed the bare `getSubmitter` as the recursive `getSubmitterFn`, defaulting `additionalSubmitterFactories` to an empty map, so a wrapping submitter (e.g. `interchainAccount` or `timelockController`) could not resolve a nested submitter whose type was only registered via a custom factory (such as the CLI's `file` submitter). In addition, the CLI's `ExtendedChainSubmissionStrategySchema` only permitted the `file` submitter at the top level and as an ICA `internalSubmitter`, rejecting it as a timelock `proposerSubmitter`. The recursive getter was changed to reuse the parent's `additionalSubmitterFactories`, and the CLI's extended ICA and timelock schemas were updated to allow any extended submitter (including `file`) as their nested submitter, so an `interchainAccount` submitter could write its `callRemote` to a file and a `timelockController` submitter could write its proposal to a file. The SDK also exported `ZBytes32String` and the `EvmTimelockControllerSubmitterProps` type so consumers could build the extended timelock schema.
